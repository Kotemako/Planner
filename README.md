## План автоматизированного тестирования формы заявки записи на курсы на сайте “Нетология”

### Перечень автоматизируемых сценариев:

#### Запись на курсы с главной страницы

Входные данные:
1. Имя (Валидное значение на Кирилице)
2. Телефон (10 цифр начинающихся на +7)
3. Электронная почта (Валидное значение на латинице с обязательными атрибутами "@" и "." в названии)

Шаги:
1. Зайти на сайт Netology.ru
2. На главной странице выбрать блок “Программирование”
3. Далее на открывшейся странице со списком курсов выбрать “Тестировщик ПО”
4. Нажать на этот блок
5. После загрузки страницы нажать на кнопку "Записаться" (или промотать страницу вниз до конца)
6. Ползунок опуститься в нижнюю часть страницы
7. Вбить валидные данные: "Имя", "Телефон" и "Электронная почта"
8. Нажать на кнопку "Записаться"

Ожидаемый результат:
Перенаправление на страницу с подтверждением валидности данных и отправка их в базу данных сайта.
Подтверждение от менеджеров, что запись прошла успешна

### Перечень используемых инструментов:
- JDK 11. Код тестов на Java.
- IntelliJ IDEA. Среда подготовки авто-тестов.
- Github. Как репозиторий для хранения кода.
- JUnit Jupiter. Фреймворк для тестирования.
- Gradle. Система сборки кода.
- Faker. Плагин для генерации данных.
- Lombok. Плагин для минимизации строк кода за счет аннотаций.
- Selenide. Продвинутый плагин для тестирования веб-интерфейса.

### Перечень необходимых разрешений/данных/доступов:
1. Зарегистрированный пользователь (данные для входа в ЛК) на портале netology.ru с заполненным профилем(имя, телефон, электронная почта)
2. Требования к полям формы: валидные варианты заполнения каждого поля
3. Возможность отправки более одного раза формы с идентичными данными
4. Доступ к базе данных сайте netology.ru
5. Разрешение от владельца сайта проводить автоматизацию по заданной форме
6. Доступ администратора от разработчиков сайта для внедрения API сайта

### Перечень и описание возможных рисков при автоматизации:
1. Наличие точной и полной спецификации с требованиями к полям формы и структуре сайта в целом.
2. Изменение структуры и верстки страниц перехода к форме записи, а также ее полей. При даже незначительных изменениях возможно падение тестов и дальнейшая доработка.
3. Изменение базы данных в ходе выполнения проекта
